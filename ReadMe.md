# 커피 한잔할래요~

## 0. 문제 해결 전략 수립

### 설계 내용

본 프로젝트는 커피 주문에 필요한 핵심 기능을 기준으로 도메인을 분리했다.

주요 도메인은 다음과 같다.

| 도메인 | 설명 |
|---|---|
| Member | 사용자 식별 정보 |
| CoffeeMenu | 커피 메뉴 정보 |
| Point | 사용자 포인트 잔액 |
| PointHistory | 포인트 충전/사용 이력 |
| CoffeeOrder | 커피 주문/결제 내역 |

API는 필수 요구사항에 맞춰 아래 4개를 구현했다.

| 기능 | Method | URL |
|---|---:|---|
| 커피 메뉴 목록 조회 | GET | `/api/menus` |
| 포인트 충전 | POST | `/api/points/charge` |
| 커피 주문/결제 | POST | `/api/orders` |
| 인기 메뉴 조회 | GET | `/api/menus/popular` |

---

### 설계의 의도

단순히 API를 동작시키는 것보다, 다수 서버 환경에서도 데이터가 꼬이지 않는 구조를 목표로 설계했다.

포인트는 결제에 직접 사용되는 중요한 데이터이므로 `Member`에 단순 컬럼으로 두지 않고 `Point` 엔티티로 분리했다.

또한 포인트가 충전되거나 사용될 때마다 `PointHistory`에 이력을 저장하여 잔액 변경 내역을 추적할 수 있도록 했다.

주문은 `CoffeeOrder`에 저장하고, 인기 메뉴는 주문 테이블의 결제 완료 데이터 기준으로 집계했다.

이를 통해 서버 메모리나 임시 캐시 값이 아니라 실제 주문 데이터를 기준으로 정확한 주문 횟수를 계산할 수 있다.

---

### 선택한 문제 해결 전략 및 분석 내용

포인트 충전과 주문 결제는 동시에 요청이 들어오면 잔액이 중복 차감될 수 있다.

예를 들어 사용자가 `4,500P`를 가지고 있을 때 `4,500원`짜리 커피 주문이 동시에 2번 들어오면, 둘 다 같은 잔액을 보고 결제에 성공하는 문제가 발생할 수 있다.

이를 방지하기 위해 포인트 조회 시 `PESSIMISTIC_WRITE` 락을 사용했다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select p from Point p where p.member.id = :memberId")
Optional<Point> findByMemberIdWithLock(@Param("memberId") Long memberId);
```

이 방식은 같은 사용자의 포인트 row를 하나의 요청이 수정하는 동안 다른 요청이 동시에 수정하지 못하게 막는다.

또한 주문/결제 로직은 `@Transactional`로 묶었다.

포인트 차감, 주문 저장, 포인트 사용 이력 저장은 하나의 작업처럼 함께 성공하거나 함께 실패해야 하기 때문이다.

### 기술적 선택 이유

| 기술/방식 | 선택 이유 |
|---|---|
| Spring Boot | REST API를 빠르게 구성하고 계층형 구조로 구현하기 적합 |
| Spring Data JPA | Entity 중심으로 DB 접근을 단순화하고 트랜잭션 관리가 쉬움 |
| MySQL | 포인트, 주문, 이력처럼 정합성이 중요한 데이터를 저장하기 적합 |
| `@Transactional` | 포인트 차감, 주문 저장, 이력 저장을 하나의 트랜잭션으로 처리하기 위해 사용 |
| `PESSIMISTIC_WRITE` | 다수 서버 환경에서 같은 포인트 row가 동시에 수정되는 것을 방지 |
| JPQL `@Query` | 최근 7일 인기 메뉴를 주문 테이블 기준으로 집계하기 위해 사용 |
| Docker Compose | app1, app2, nginx, mysql 구성으로 다수 인스턴스 환경을 로컬에서 검증하기 위해 사용 |
| MockDataPlatformClient | 실제 외부 플랫폼 대신 주문 데이터 전송 요구사항을 테스트 가능한 형태로 구현 |

## 동시성 테스트 결과

동일한 사용자가 4,500P를 보유한 상태에서 4,500원짜리 메뉴를 동시에 2번 주문하는 상황을 테스트했다.

포인트가 1잔 가격만 존재하므로 정상적인 경우 2개 요청 중 1개만 성공해야 한다.

테스트 결과는 다음과 같다.

| 항목 | 결과 |
|---|---:|
| 동시 주문 요청 수 | 2 |
| 성공한 주문 수 | 1 |
| 실패한 주문 수 | 1 |
| 최종 포인트 잔액 | 0P |
| DB에 저장된 주문 수 | 1 |

이를 통해 같은 사용자의 주문 요청이 동시에 들어와도 포인트가 중복 차감되지 않고, 주문도 1건만 생성되는 것을 확인했다.
